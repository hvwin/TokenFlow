# 项目约束

## 1. 目标边界

- 项目目标是让 TokenFlow 成为可持续吞噬猎物能力源的闭合智能体能力生态。
- 不把项目做成单一宿主专属实现。
- 不把 prompt 策略和真实执行混在一个层里。

## 2. 实现边界

- `core` 只做纯能力逻辑和路由决策。
- `adapters` 只做宿主对接，不承载业务真源。
- `templates` 只做生成模板，不存运行时状态。
- `manifests` 只做机器可读声明，不写执行副作用。

## 3. 接入优先级

1. `MCP`
2. `Skill`
3. `Hook`
4. `Command`

说明：

- 只要宿主支持 `MCP`，优先用 `MCP` 做真实执行。
- 只要宿主支持 `Skill`，用它承载语义触发和 SOP。
- 宿主支持 `Hook` 时，可叠加前置/后置治理。
- `Command` 永远作为显式兜底入口保留。

## 4. 禁做项

- 不做单一 IDE 私有绑定。
- 不做自动覆盖用户现有接入配置的行为。
- 不在未确认情况下删除文件或重置目录。
- 不把 generated adapter 当成唯一真源。

## 5. 猎物层约束

- `prey/` 只记录能力真源、映射和吸收状态，不承载运行时副作用。
- 首批工具只是样本，不是上限。
- 每个新猎物都必须能单独读懂当前吸收阶段。
- 候选猎物优先按能力家族入池，不按工具名碎片化入池。
- 猎物层与 core 层分离，避免把分析文档写成实现代码。
